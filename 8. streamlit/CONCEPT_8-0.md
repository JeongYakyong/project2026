## 1. 컨셉 (한 줄)

**"신재생이 만든 잔여부하를 가스 발전이 메운다 — 그 체인(수요 → 신재생 → net_load → 가스)을 매일의 예측으로 보여주는 발전사업자용 브리핑 대시보드."**

- 정확도 자랑 페이지가 아니라 **명제 입증(§1.2)을 시각적으로 전달**하는 페이지다.
- 차별점 = ① 5→6→7 실제 서빙 체인이 살아서 돈다 ② 실측 대비 오차를 숨기지 않는다(정직성 §5.4) ③ brief_ai가 수치를 자연어 브리핑으로 바꿔준다(공모전 가점 의도).

## 2. 목표 (공모전 심사 관점)

1. **명제가 한 화면에서 읽힌다**: 신재생↑ → net_load↓ → 가스↓ 의 연결을 차트 스택으로 보여준다.
2. **살아있는 예측**: 기준일(origin)을 고르면 D+1~D+7(가스는 D+1~D+12)의 예측 곡선과, 지난 날짜는 실측 비교까지 나온다.
3. **KOGAS 연결(자격 앵커 §2)**: 가스 발전량(MWh) → 송출량(TON, ×0.1521) → 가스비(원) 환산을 화면에 명시한다.
4. **정직성**: 단계별 MAPE, 체인 손실(ORACLE 10.8% vs 체인 ~13%), 자기참조 구간 "추정" 표기를 검증 탭에 그대로 둔다.
5. brief_ai 브리핑(3~5문장)으로 "그래서 발전사업자는 뭘 하면 되는가"를 답한다.

## 3. 디자인

### 3.1 사이드바
- **지역**: 전국 / 제주 (라디오)
- **기준일(origin)**: 날짜 선택 — 데이터 보유 기간으로 제한(§6.3)
- **지평**: D+1 ~ D+7 슬라이더 (가스는 D+12까지)
- (관리자) 캐시 초기화

### 3.2 탭 구조 (§6.1 골격 유지 + 보완 2건)

| 탭 | 내용 | 데이터 출처 |
|---|---|---|
| **Tab 1. 개요** | 명제 한 장 다이어그램(수요−신재생=net_load→가스), 체인 구조 5→6→7, 데이터 귀속(KOGAS 앵커) | 정적 |
| **Tab 2. 예측 대시보드 (핵심)** | 상단 brief_ai 카드 → 차트 스택: 기상 개황 / 수요 vs 신재생 / net_load / **가스 송출량(TON)+가스비** → 하단 시간대별 테이블. 지난 날짜는 실측 오버레이 | forecast 테이블: `est_demand_land`·`est_market_renew_land`·`est_net_load_land`·`est_gas_gen_land`·`est_gas_sendout_ton_land` (제주: `jeju_est_demand_lh`·`est_*_jeju_lh`·`est_net_load_jeju`·SMP) |

**보완 ①(제주 분기)**: 제주는 가스 서빙이 없다(7-B는 명제 probe). 제주 선택 시 Tab 2의 마지막 카드는 가스 대신 **SMP D+1/D+2 + 음수가격 경보**(4단계, ROC-AUC 0.973)를 보여준다. → 제주=시장(SMP) 관점, 전국=가스 관점으로 지역별 결말이 자연스럽게 갈린다.

**보완 ②(SMP 명시)**: 기존 §6.1엔 4단계 SMP가 빠져 있었다. 위처럼 제주 Tab 2에 편입하는 것으로 반영한다.

### 3.3 brief_ai
- 입력: 당일 예측 요약(기상·수요·신재생·net_load·가스 또는 SMP, 전일 실측 대비)을 작은 JSON으로.
- 출력: 3~5문장 — 기상 개황 / net_load 변동 / 가스(또는 SMP) 동향 / 발전사업자 액션 제안.
- 같은 (날짜, 지역)은 24시간 캐시. API 실패 시 "AI 브리핑 일시 사용 불가" + 차트는 그대로(§6.2).

## 4. 기술 골격 (가볍게)

- 단일 `app.py` + `pages/` 없이 탭으로(데모 규모에 중첩 불필요). 차트는 plotly.
- DB는 **읽기 전용**: `1. data_fetcher_and_db/data/input_data_{jeju,land}.db` 의 forecast·historical만 SELECT. 데모 앱에서 서빙 스크립트를 직접 실행하지 않는다(예측은 사전 적재된 값을 보여주는 방식 권장 — 아래 G-15 ③).
- `st.cache_data(ttl=...)`로 쿼리 캐시. 예측 실행 버튼을 둔다면 IP당 횟수 제한(§6.3).

## 5. 결정 필요 — G-15 후보 (구현 전 확정, §0.3)

> **2026-06-10 확정(PROJECT.md §7 G-15)**: ① 자체 서버 호스팅(로컬 DB 실시간 읽기) ② Gemini API 유지 ③ 사전 적재 기본 + 시연용 실행 버튼 병행 ④ 표시 기간 = 데이터 보유 범위(전국 2026-02~, 제주 2025-12~) ⑤ SMP 데모에서 일단 제외(제주는 net_load까지).

1. **배포 위치 + DB 접근**: Streamlit Community Cloud는 로컬 sqlite 파일 동기화가 안 됨. 후보 = (a) 주기적 DB 스냅샷을 repo/스토리지에 푸시 (b) 자체 서버(집/클라우드 VM)에서 streamlit 직접 호스팅 (c) 데모용 정적 추출(parquet) 동봉. **앱 코드보다 이 결정이 먼저다.**
2. **brief_ai LLM**: 기존 명세는 Gemini API(§6.2). 유지/변경 + API 키·비용 확인.
3. **예측 갱신 방식**: 실시간 실행 버튼 vs **사전 적재(서빙 3종을 cron으로 5→6→7 순서 실행, 앱은 읽기만)**. 사전 적재 권장 — PROJECT.md 지적사항(체인 자동화 공백)도 같이 해소된다.
4. **표시 기간**: 백필을 언제부터 보여줄지(자기참조 구간 "추정" 표기 포함).
5. **SMP 포함 여부**: 위 보완 ① 제안대로 제주 탭에 편입할지 최종 확정.
