# 작업의뢰서 — LNG 조달 시나리오 재설계 (v2)

> 대상: `7. land_gas_forecaster/` (또는 조달 시나리오 폴더)
> 기존 산출물: `lng_procurement_sim.py`, `REPORT_procurement.md` (이 둘을 v2로 개편)
> 한 줄 목적: **발전용 가스수요 예측모델(우리)이 naive 대비 LNG 조달 의사결정에서 주는 가치를, 정직하게(조작 없이) 보인다.**

---

## 0. 이 작업의 전제 (먼저 읽을 것)

- 이건 "우리 모델이 이기는 시나리오 만들기"가 **아니다.** 결과를 정해놓고 거기 닿는 설정을 고르면 cherry-pick이고, 공모전(공공데이터·재현 가능)에서 실격 사유다. PROJECT.md §5.4 "정직성 = 강점" 원칙을 따른다.
- **모든 모델에 정책·하한·가격·프리미엄·안전재고를 동일하게** 주고, **오직 소비 예보만 다르게** 한다. 그래야 비용 차이를 "예보 정밀도" 하나로 정직하게 귀속할 수 있다.
- 결과가 작게 나오면 그것도 정직한 결과다. 작게 나와도 "예보 정밀도 → 비상구매 회피"라는 메커니즘 자체가 인사이트다.

## 1. ★ 착수 전 확인 게이트 (이게 통과 안 되면 시작하지 말 것)

### 게이트 A — D+15 지평 가용성
- 리드타임을 **14일**로 둘 거라, 보호구간(protection interval) = lead + review = **15일**이다.
- 따라서 `est_horizon_land` 테이블(또는 우리 모델 송출 예측 소스)에 **각 base_date에서 D+1 ~ D+15까지 `est_gas_sendout_ton_land`(또는 발전량×0.1521)가 실측 예측으로 존재해야 한다.**
- **확인 작업**: `est_horizon_land`의 `horizon_d` 분포를 출력하라. 최대 지평이 D+15 미만이면(예: D+7, D+12) 아래 중 하나를 사용자에게 보고하고 결정받는다.
  - (a) 리드타임을 실제 최대 지평−1로 낮춘다(예: D+12 가용 → 리드 11일).
  - (b) D+12까지만 쓰고 D+13~15는 명시적으로 "기후값 폴백"임을 리포트에 정직하게 적는다(권장하지 않음 — 비교 오염).
- **현재 폴백 버그**: 기존 `f_ours(t,k)`는 지평 값이 없으면 조용히 `avg_daily`로 떨어진다. 이러면 우리 모델이 장지평에서 기후값이 돼 비교가 무효가 된다. **폴백이 발생하면 카운트해서 출력하고, 폴백 비율이 0이 아니면 경고를 띄울 것.**

### 게이트 B — JKM 가격 소스
- 코드가 읽는 `second_dataset/LNG Japan_Korea Marker PLATTS Future daily.csv`의 **실제 기간과 출처를 확인**하라. 기존 리포트는 "1~2월 저가 → 3월 스파이크 22.35"를 묘사하지만, 사용자가 가진 검증용 JKM은 **2026-04~06 평탄 고가(15~20)**다.
- **결정(사용자 확정)**: 가격은 **실측 spot 시계열 그대로** 쓴다(합성·가공 금지). 평탄해도 무방 — Framing B의 비용 격차는 가격 곡선 모양이 아니라 "비상물량 × spot × 프리미엄"에서 나오므로 평탄 가격에서도 성립한다.
- JKM 파일이 합성으로 의심되면 사용하지 말고 사용자에게 보고할 것.

## 2. 파라미터 (days-of-supply 단위로 통일)

| 항목 | 정의 |
|---|---|
| `daily_max` | 평가기간 일 소비량의 **최대치**. 일 소비 = `historical.gen_gas_kr`(MW, 시간별)를 일 합산 × `0.1521 ton/MWh` (G-5) |
| FLOOR (하한·비상 트리거) | **14 × daily_max** (리드타임을 peak 수요로도 버티는 보수적 하한) |
| START (현재재고) | **21 × daily_max** |
| CAP (상한·저장 한계) | **30 × daily_max** ※ 소비일수로 잡은 단순화. 리포트에 "실제 저장은 물리 탱크 용량" 한 줄 명시 |
| LEAD (리드타임) | **14일** (게이트 A 결과에 따라 조정 가능) |
| 보호구간 | LEAD + 1 = **15일** (= 예보가 커버해야 하는 LTD 구간) |
| PREMIUM (비상 현물 프리미엄) | **1.25** (양쪽 동일) |
| MMBTU_PER_TON | 52.0 |

- 단위변환·G-5 계수(0.1521)는 기존 코드 그대로.
- `daily_max`를 코드에서 실측으로 계산해 출력할 것(하드코딩 금지).

## 3. 시뮬레이션 설계 — Framing B (동일 안전재고)

### 3.1 비교 모델 (소비 예보만 다름, 나머지 전부 동일)
- `oracle` : 실측 소비(완벽 예보, 상한 기준선)
- `ours` : `est_horizon_land`의 (base_date, k) 송출 예측
- `naive` : 정직한 주간 lag (target − 168h 등, 기존 `f_naive` 방식 유지)

### 3.2 정책 (3모델 공통)
- base-stock(재고포지션 = 보유 + 운송중) 정책 유지.
- **안전재고(SS)를 세 모델에 동일한 값으로 고정**한다. (A 프레이밍처럼 모델별 σ로 다르게 주지 말 것 — B의 핵심.)
- 발주량 q = max(0, S − 재고포지션), 도착은 t+LEAD. 발주일 가격으로 비용 고정.
- 하한(FLOOR) 붕괴 시: 부족분을 **그날 spot × PREMIUM**으로 급매수해 FLOOR 복구. 붕괴 1회로 카운트.

### 3.3 ★ 핵심 산출물 — 신뢰도 곡선 (안전재고 sweep)
> B가 "naive만 붕괴하도록 SS를 맞췄다"는 의심을 원천 차단하는 장치. **단일 SS 값이 아니라 전 구간을 쓸어본다.**

- **SS를 0일치 ~ 14일치(daily_max 기준)까지 sweep**하며 각 SS에서 세 모델을 모두 시뮬레이션.
- 출력 곡선 (모델 3개 동시):
  - **그림 1 — 신뢰도 곡선**: x축 = 보유 안전재고(일수), y축 = 붕괴 횟수. (또는 y = 비상비용 $)
  - 우리 모델 곡선이 항상 oracle과 naive **사이**에 깔리는지 확인. 한 점이 아니라 **전 구간 우위**를 보이는 게 목적.
- **세로로 읽으면 B**(같은 SS에서 붕괴/비상비용 격차), **가로로 읽으면 A**(같은 붕괴0에 필요한 SS 격차) → 한 장에 둘 다 들어감.

## 4. 정직성 가드레일 (리포트에 반드시 명시)

- 대표 SS 값은 **결과를 보기 전에** 정책 근거로 고정했음을 명시(또는 sweep 곡선으로 대체).
- 단일 5개월·단일 JKM 경로 시나리오 → 절대값 일반화 금지, **모델 간 상대 비교**가 요지.
- CAP를 소비일수로 잡은 단순화임을 명시.
- 폴백(기후값) 발생 비율을 리포트에 적을 것(0이어야 정상).
- naive는 "정직한 주간 lag"이지 일부러 멍청하게 만든 게 아님을 명시.
- 가격은 실측 spot 그대로(합성 금지).
