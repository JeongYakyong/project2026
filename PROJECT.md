# 제주 전력시장 예측 데모 — 프로젝트 마스터 문서

> **이 문서의 용도**
> 1. 본인이 진행상황을 추적하기 위한 단일 출처(Single Source of Truth)
> 2. Claude 대화 시 컨텍스트로 첨부 → 매번 배경 설명 반복 방지
> 3. Claude Code 사용 시 AI 참고 문서로 제공 → 작업 일관성 유지
>
> **갱신 규칙**: §0~§4는 고정. §5(체크리스트)와 §7(진행 로그)만 수시 업데이트.

---

## 0. 프로젝트 개요

### 0.1 한 줄 정의
기상청·전력거래소 공공데이터를 활용해 **제주 지역 전력수요(est_demand) → net_load → SMP를 3단으로 +24/+48h 예측**하고, 발전사업자에게 유용한 브리핑을 제공하는 Streamlit 기반 데모 서비스.

### 0.2 출품 대상
제14회 산업통상부 공공데이터 활용 아이디어 공모전 — **제품 및 서비스 부문**

### 0.3 마감일
**[YYYY-MM-DD]** ← 확정되면 채우기. 이 날짜 기준 D-3까지 제출 완료.

### 0.4 메인 셀링 포인트 (우선순위 순)
1. **제주 지역에서 동작하는 완성된 데모** — 실제 URL 접근 가능, 예측 결과와 실제값 비교까지 표시
2. **3단 예측 구조의 차별성** — est_demand(전력수요) → Cross-Attention PatchTST(net_load) → LGBM(SMP)
3. **AI 브리핑(brief_ai)** — Gemini API 기반, 예측 결과를 자연어 인사이트로 변환
4. **전국 확장 가능성** — 동일 구조를 전국에 적용한 "맛보기" 결과 (Track B)

### 0.5 명시적 비목표 (NOT goals)
- **전국 단위 SMP 예측 — 시장 구조상 불가**
  - 제주: 전력거래 시범사업 적용 지역. 실시간 시장(SMP) 운영 중 → SMP 예측의 의사결정 가치 존재
  - 전국: 시범사업 미적용. 실시간 시장 구조 부재 → SMP 예측 자체가 의미를 가지지 못함
  - 따라서 전국은 **net_load 예측까지만** 수행. SMP 예측을 "안 한 것"이 아니라 "할 이유가 없는 것"
- 상용 서비스 수준의 보안·확장성 (데모 수준 OK)
- 18개 전체 지역 커버 (1~3개 샘플로 충분)
- 모델 성능 SOTA 달성 (이미 동작하는 수준 유지, 하이퍼파라미터 미세 조정만)

### 0.6 프로젝트 컨텍스트 메모 (다음 대화로 가져갈 맥락)

이 문서를 처음 보는 Claude/Claude Code가 놓칠 수 있는, 본인 외엔 모르는 배경 정보들.

**개발자(본인)의 작업 스타일**
- 단계별로 하나씩 구동하면 결과는 낼 수 있음. 즉, **개별 기술 역량은 충분**.
- 약점은 "**여러 단계를 이어가는 흐름 관리**". 의사결정이 안 된 상태에서 무거운 작업을 시작하면 흐름이 끊김.
- 따라서 AI 협업 시 "**작업 쪼개기 + 의사결정 게이트 명시**"가 핵심. 큰 덩어리로 던지지 말 것.

**과거에 막혔던 지점 (반복 방지용)**
- Track B(전국 확장)를 결정 못 한 채로 18개 지역 선정, ASOS/KPX 수집 같은 무거운 작업을 먼저 시작 → 흐름 정지
- 교훈: **"이걸 해야 하는가?"가 결정되기 전에는 작업 시작 금지**. 항상 Decision Gate(§5)를 먼저 통과시킬 것.

**brief_ai의 진짜 목적**
- 단순 기능이 아니라 **공모전 가점 유도가 명시적 의도**. 우선순위 판단 시 이 점 반영.
- "Gemini로 그냥 요약" 수준이 아니라, 발전사업자에게 의미 있는 자연어 인사이트가 나와야 가점 효과 있음.
- 프롬프트 엔지니어링에 시간 투자 가치 있음 (Week 2).

**Track C(공모전 제출물)에서 본인이 강조한 부분**
- "**필요한 스크린샷·테이블을 미리 정리**"하는 게 중요. Week 4에 몰아서 만들면 늦음.
- Week 2~3에서 데모 동작 시 **나중에 쓸 스크린샷을 그때그때 캡처**해두는 습관 필요.
- §4.5 스크린샷 패키지 항목을 Week 4 직전이 아니라 **Week 2부터 채워나가야** 함.

**전국 데이터 수집의 위치**
- "Track B 진행 여부와 별개로, 데이터 수집 가능성 조사(B1)는 어차피 함" — 본인 의사
- B1 결과는 Track B를 안 하더라도 **기획서 §4.4 확장 로드맵의 근거 자료**로 활용됨
- 따라서 B1만큼은 Week 2에 반드시 완료할 것

### 1.2 Track B — 전국 확장 가능성 (보너스)

**시장 구조 전제**: 제주는 전력거래 시범사업 적용 지역으로 실시간 시장(SMP)이 운영되지만, 전국은 시범사업 미적용으로 SMP가 의사결정 변수가 아님. 따라서 전국은 **net_load까지만** 예측하며, 이는 모델 한계가 아니라 **시장 구조에 맞춘 의도된 설계**.

| ID | 산출물 | 형태 | 완료 기준(DoD) |
|----|--------|------|----------------|
| B1 | 전국 데이터 접근 가능성 조사 | 1페이지 메모 | KPX 발전량·ASOS API 조건·한도 명세, 18→N개 축소안 |
| B2 | 샘플 지역 fetcher | Python 코드 | 1~3개 지역 데이터 7일치 수집 |
| B3 | 전국 샘플 net_load 예측 | 결과 표/차트 | 7일 결과로 "가능성" 입증. SMP는 시장 구조상 비대상 |
| B4 | Streamlit "전국 확장 미리보기" 탭 | 페이지 | B3 결과 시각화 + 시장 구조 차이 설명 문구 포함 |

**Track B 원칙**: "구현"이 아니라 "가능성 입증". 제주 모델과 동등할 필요 없음.

## 3. Streamlit 데모 사양서

### 3.1 페이지 구조
```
[사이드바]
- 날짜 선택 (캘린더, 데이터 보유 범위 내)
- 지역 선택 (제주 / 전국 샘플) ← B4 완성 시 활성화
- "예측 실행" 버튼
- (관리자) 캐시 초기화 버튼

[메인 영역 — 탭]
Tab 1. 개요 (Landing)
  - 프로젝트 한 줄 설명
  - 모델 구조 다이어그램 (어떻게 동작하는지)
  - "지금 시연 보기" CTA

Tab 2. 예측 대시보드 (핵심)
  - 상단: brief_ai 카드 (3~5문장 자연어 브리핑)
  - 중단: 차트 4개 (날씨 / est_demand 예측 vs 실제 / net_load 예측 vs 실제 / SMP 예측 vs 실제)
  - 하단: 수치 테이블 (시간대별 예측·실제·오차)

Tab 3. 모델 검증 (정확도)
  - MAE/MAPE 지표
  - 최근 N일간 예측 오차 추이
  - 모델 구조 설명 (PatchTST + LGBM 2단 구조)

Tab 4. 전국 확장 미리보기 (Track B 완성 시)
  - 샘플 지역 N개 net_load 예측 결과
  - "가능성 입증" 메시지
  - **시장 구조 차이 설명 박스**: "전국은 전력거래 시범사업 미적용 지역으로 SMP 예측 비대상.
    net_load 예측만으로도 발전사업자의 출력 계획 수립에 직접 활용 가능"
```

### 3.2 데이터 흐름
```
[crontab, 백그라운드, 사용자 접근 불가]
매일 06:00 → fetch_weather() + fetch_power_data()
매일 06:30 → predict(today):
              fetch → est_demand → net_load → SMP
              → 결과 캐시 저장

[Streamlit 사용자 접근]
사용자가 날짜 선택 → 캐시에서 로드 → 표시
"예측 실행" 버튼 → rate limit 확인 → 캐시 우선, 없으면 실시간 실행
```

### 3.3 brief_ai 사양
- **입력**: predict(date) 결과 (날씨·est_demand·net_load·SMP 예측값, 실제값, 통계)
- **프롬프트 구조**: 시스템 프롬프트(역할 정의) + 데이터 요약(JSON) + 출력 형식 지시(3~5문장)
- **출력 항목 후보**: 기상 개황 / 전력수요(est_demand) 특이점 / net_load 변동 / SMP 동향 / 발전사업자 액션 제안
- **모델**: Gemini API (예: gemini-2.x flash)
- **에러 처리**: API 실패 시 "AI 브리핑 일시 사용 불가" + 차트는 그대로 표시

### 3.4 운영 정책 (중요)
| 항목 | 정책 | 근거 |
|------|------|------|
| 데이터 수집 | crontab 백그라운드, 사용자 트리거 불가 | 기상청·KPX API 한도 보호 |
| 예측 실행 | 캐시 우선. 캐시 없을 때만 실시간 실행 | LGBM/PatchTST 추론 비용 |
| "예측 실행" 버튼 | IP당 시간 N회 제한 (예: 3회) | 무한 클릭 방지 |
| brief_ai 호출 | 같은 date+region 조합은 24h 캐시 | Gemini API 호출 비용 |
| 캐시 초기화 | 관리자 페이지 또는 환경변수 인증 | 일반 사용자 접근 금지 |
